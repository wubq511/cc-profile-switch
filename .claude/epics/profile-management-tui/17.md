# Task: Core Profile Management Services

## Description

Implement the shared core services for V0.2 profile lifecycle management. These services must own the filesystem and config mutations for copying, renaming, removing, listing, and default profile management so CLI and TUI behavior cannot diverge.

## Acceptance Criteria

- [ ] Core exposes service functions for profile list summaries, copy, rename, remove, get default, set default, clear default, and launch profile resolution.
- [ ] `copyProfile` validates the source profile, refuses existing targets, copies profile content, updates `profile.json.name`, `createdAt`, `updatedAt`, and repairs `autoMemoryDirectory`.
- [ ] `renameProfile` validates the source, refuses existing targets, moves the profile directory, updates manifest metadata, repairs `autoMemoryDirectory`, and updates matching `defaultProfile` / `lastUsedProfile` references.
- [ ] `removeProfile` refuses deletion unless confirmation exactly equals the profile name, creates a timestamped backup before deletion, deletes only the selected profile, and clears matching config references.
- [ ] Removing the default profile and removing the last profile are both supported and leave `config.json` valid.
- [ ] Unit tests cover successful paths, invalid names, missing profiles, validation failures, target collisions, wrong confirmation, and config reference updates.

## Technical Details

- Prefer a new focused module such as `src/core/profile-management.ts` or a clearly bounded extension to `src/core/profile.ts`.
- Reuse `getProfileTemplatePaths`, `validateProfile`, `isLaunchBlocking`, `loadAppConfig`, `saveAppConfig`, `writeJsonFile`, and Windows path helpers.
- Keep all path writes inside `%USERPROFILE%\.cc-profile-switch`.
- Update `claude-home\settings.json` only for fields required to repair profile-local path references.
- Do not read from, copy from, or write to the real `C:\Users\h\.claude`.
