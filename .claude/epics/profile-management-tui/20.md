# Task: TUI Flow Controller

## Description

Create a testable TUI controller that models the profile management flow independently from terminal rendering. The controller should call shared core services and expose a compact action loop suitable for `ccps tui`.

## Acceptance Criteria

- [x] A `src/tui` controller layer can list profiles with validation status, default marker, last-used marker, and description.
- [x] The controller supports copy, rename, remove, set default, clear default, show details, validate, and launch dry-run actions.
- [x] Removal uses the same exact-name confirmation rule as the CLI.
- [x] The controller handles zero profiles by showing create/init guidance rather than crashing.
- [x] Controller tests use mocked prompt/output ports and mocked or temporary core services.
- [x] No filesystem mutation rules are implemented in the TUI controller beyond calling shared services.

## Technical Details

- Define narrow ports such as `select`, `input`, `confirmByName`, and `writeOut`.
- Keep controller output operational and compact.
- Treat launch from TUI as dry-run only for V0.2.
- Avoid coupling controller tests to a specific prompt library.
